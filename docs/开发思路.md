# 实时背景替换项目 — 开发思路与问题解决

## 一、项目目标

实现一个**实时视频背景替换**桌面应用：摄像头捕获画面 → 人像分割 → 背景替换 → 实时显示。要求流畅、低延迟、边缘自然。

## 二、整体开发思路

### 第一阶段：最小可用版本（MediaPipe）

**思路**：先用最简单的方案跑通完整流程，再逐步优化。

1. **选择 MediaPipe Selfie Segmentation** 作为初始算法
   - 优点：Google 出品，开箱即用，CPU 极快，无需训练
   - 缺点：分割精度一般，边缘较粗糙
2. **GUI 构建**：当前代码使用 tkinter，负责视频显示、背景选择、截图录屏
3. **多线程架构**：视频采集和处理放在子线程，避免 GUI 卡顿

> **关键决策**：先跑通再优化，不要一开始就追求完美的分割效果。

### 第二阶段：引入 MODNet 提升质量

**思路**：MediaPipe 效果不够好，引入专业的人像抠图模型。

1. **选择 MODNet**：专为人像 matting 设计，输出连续 alpha matte（而非二值 mask）
2. **工厂模式**：设计 `BackgroundChangerFactory`，通过 `ALGORITHM_ID` 切换算法
3. **延迟导入**：PyTorch 和 MediaPipe 的 DLL 冲突，必须按需加载

> **关键决策**：用工厂模式解耦算法和 GUI，新增算法只需加一个 segmenter。

### 第三阶段：引入 RVM 优化视频体验

**思路**：MODNet 是为静态图片设计的，视频场景需要时序一致性。

1. **选择 RVM (RobustVideoMatting)**：内置循环状态，天然适合视频
2. **ONNX Runtime 推理**：避免 PyTorch 依赖，CPU 上 40+ FPS
3. **即插即用**：得益于工厂模式，新增算法只需写一个 segmenter 文件

> **关键决策**：不同场景选不同工具——静态图 MODNet，实时视频 RVM。

### 当前阶段：以 RVM 作为上位机默认算法

**思路**：静态离线合成测试曾让 MOG2 看起来很强，但直播压力测试显示它对开播即有人、光照变化、压缩噪声和摄像头抖动更敏感。补入 PPM-100 真实人像 matte 后，RVM 在综合分、关键风险场景分和实时速度之间最均衡。

1. **默认算法切换为 RVM**：`config/constants.py` 中 `OPTIMAL_ALGORITHM_ID = 2`，`OPTIMAL_ALGORITHM_NAME = "RVM"`。
2. **保留多算法工厂**：显式传入算法 ID 仍可复现实验，GUI 上位机默认只显示当前最佳算法。
3. **数据驱动选型**：以 `docs/reports/直播场景模拟测试报告.md` 作为默认算法决策依据，而不是只看静态 benchmark。

> **关键决策**：默认上位机服务实时直播，因此优先选 RVM；MOG2 保留为无模型、空背景预热充分时的备选基线。

## 三、问题解决思路

### 思路 1：分层隔离，逐步排查

**案例：WinError 1114 DLL 冲突**
这个问题最棘手，表现为 PyTorch 的 `c10.dll` 加载失败，但错误信息只有一个笼统的 DLL 初始化失败。

**排查步骤**：

1. **隔离测试**：单独 `import torch` 正常，说明 torch 本身没问题
2. **缩小范围**：逐行注释其他 import，定位到 PyQt5 导入后 torch 就失败
3. **查资料**：发现 PyQt5 会修改 Windows DLL 搜索路径
4. **验证假设**：调换 import 顺序，先 torch 后 PyQt5，问题解决
5. **进一步隔离**：MediaPipe 和 PyTorch 也冲突 → 延迟导入

**总结**：遇到 DLL/导入问题，**隔离 → 最小复现 → 逐步复原**。

### 思路 2：对齐官方实现

**案例：MODNet 模型加载失败**
自定义的模型定义（ResNet50）与预训练权重（MobileNetV2）不匹配。

**排查步骤**：

1. **看报错**：`load_state_dict()` 提示 key 不匹配
2. **对比 key**：打印模型和权重的 key，发现完全不同
3. **找官方源码**：GitHub 上找到 MODNet 官方实现，backbone 是 MobileNetV2
4. **替换模型定义**：直接使用官方代码
5. **处理 DataParallel 前缀**：权重 key 带 `module.` 前缀，strip 掉

**总结**：使用预训练模型时，**必须确保模型定义和权重完全匹配**，最安全的做法是直接用官方代码。

### 思路 3：可视化中间结果

**案例：MODNet 分割质量差**
背景替换后人物半透明，但不清楚是 matte 质量问题还是混合逻辑问题。

**排查步骤**：

1. **可视化 matte**：将 alpha matte 直接显示为灰度图
2. **发现问题**：matte 整体偏灰（0.3~0.7），而非理想的黑白分明
3. **检查预处理**：发现输入图像被强制 resize 到 512×512 正方形，破坏宽高比
4. **检查尺寸对齐**：MODNet encoder 多次下采样，要求 32 的倍数
5. **逐步修复**：保持宽高比 → 32 像素对齐 → 原始分辨率混合

**总结**：效果不好时，**可视化中间结果**是最有效的调试手段。

### 思路 4：量化对比，数据驱动

**案例：性能优化**
MODNet 在 CPU 上约 10 FPS，不够流畅。

**优化步骤**：

1. **测速**：`time.time()` 精确测量每个环节耗时
2. **定位瓶颈**：推理 80ms，后处理 15ms，混合 5ms → 瓶颈在推理
3. **降低分辨率**：ref_size 512→384→256，测量速度和质量变化
4. **简化后处理**：移除耗时的形态学运算
5. **最终方案**：ref_size=384 折中，或直接换 RVM（24ms/帧）

**总结**：优化前先**量化瓶颈**，避免优化不重要的部分。

### 思路 5：工厂模式应对变化

**案例：多算法共存**
项目需要支持多种分割算法，但每种算法的依赖、接口、初始化方式都不同。

**设计思路**：

1. **统一接口**：所有算法必须实现 `process_frame()`、`load_backgrounds()`、`next_background()`
2. **工厂类**：`BackgroundChangerFactory` 根据 ID 实例化对应算法
3. **延迟导入**：在工厂方法内部按需导入，避免 DLL 冲突
4. **自动回退**：算法初始化失败时自动切换到 MediaPipe
5. **配置驱动**：`constants.py` 中一个变量控制算法选择

**总结**：面对多种实现方案，用**工厂模式 + 统一接口**解耦具体实现和上层逻辑。

## 四、经验总结

| 原则 | 说明 |
| ---- | ---- |
| **先跑通再优化** | 最小可用版本 → 逐步迭代 |
| **隔离排查** | 遇到奇怪的错误，缩小范围逐步定位 |
| **可视化调试** | 中间结果看不懂就画出来 |
| **量化驱动** | 优化前先测量，数据说话 |
| **对齐官方** | 用预训练模型必须对齐官方实现 |
| **设计模式** | 工厂模式解耦，方便扩展 |
| **选对工具** | 静态图可看 MODNet，实时直播默认 RVM，传统 CV 只作无模型基线 |
