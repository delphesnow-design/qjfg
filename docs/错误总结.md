# 背景切换工具项目问题总结与解决方案

## 1. 项目概述

本项目是一个基于 RVM 与 OpenCV 的实时背景切换上位机，支持直播场景中的人物分割和虚拟背景替换。通过 tkinter GUI 提供截图、录屏、背景上传等功能；历史算法仍保留用于实验复现。

## 2. 遇到的问题及解决方案

### 2.0 虚拟摄像头输出缺失

**问题描述**：

- 原项目只能在上位机窗口中预览背景替换结果，或保存截图/录屏
- Zoom、Teams、腾讯会议等第三方软件无法直接把处理后画面作为摄像头输入
- 会议软件只能看到物理摄像头，无法无缝使用背景替换效果

**解决方案**：

- 使用 `pyvirtualcam` 将 `VideoThread` 中的 `processed_frame` 输出到系统虚拟摄像头
- 在 GUI 中新增「开启虚拟摄像头」按钮，支持运行时开启/停止输出
- 虚拟摄像头输出与预览、截图、录屏共用同一帧处理结果，保证会议软件看到的画面与上位机一致
- Windows 环境需要先安装 OBS Virtual Camera 或其他兼容虚拟摄像头后端
- 增加 `scripts/debug/check_virtual_camera_backend.py`，用于单独检查系统虚拟摄像头后端是否可用

### 2.2 OpenCV图像编解码器缺失问题

**问题描述**：

- OpenCV 4.13.0 版本缺少图像编解码器支持
- `cv2.imwrite()` 和 `cv2.imread()` 返回 `False` 或 `None`
- 导致截图无法保存，图片无法加载

**根本原因**：

```bash
支持的写入格式: []  # OpenCV无法识别任何图像格式
```

**解决方案**：
采用[多库协同与回退策略](memory://b7b23b9d-c244-40a1-a110-e50b9c889639)：

1. **图像加载双策略**：
   - 优先使用OpenCV (`cv2.imread()`)
   - 失败时回退到PIL (`Image.open()` + 格式转换)

2. **图像保存双策略**：
   - 优先使用OpenCV (`cv2.imwrite()`)
   - 失败时回退到PIL (`Image.fromarray().save()`)

3. **格式转换处理**：
   - OpenCV使用BGR格式，PIL使用RGB格式
   - 正确实现 `cv2.cvtColor(image, cv2.COLOR_BGR2RGB)` 和 `cv2.COLOR_RGB2BGR` 转换

### 2.10 PyTorch DLL 冲突 (WinError 1114)

**问题描述**：

- 在 Windows 上运行时，`import torch` 抛出 `[WinError 1114] DLL initialization failed`
- 历史版本中 MODNet 无法初始化时会回退到 MediaPipe；当前默认链路为 RVM，RVM 失败时回退到 MOG2
- 单独运行 `import torch` 正常，但在 PyQt5 之后导入就会失败

**根本原因**：

- PyQt5 加载时会修改 Windows 的 DLL 搜索路径
- 后续加载 PyTorch 的 `c10.dll` 时找不到其依赖 DLL
- MediaPipe (TensorFlow Lite) 与 PyTorch 的 DLL 也存在冲突

**解决方案**：

1. **导入顺序修复**（`main.py`）：在 PyQt5 之前预导入 torch

   ```python
   # 必须在 PyQt5 之前导入 torch
   try:
       import torch
   except ImportError:
       pass
   # 之后再导入 PyQt5
   from PyQt5.QtWidgets import QApplication
   ```

2. **延迟导入**（`algorithms/factory.py`）：在工厂方法内部按需导入算法模块，避免 MediaPipe 和 PyTorch 同时加载

### 2.11 MODNet 模型结构不匹配

**问题描述**：

- 自定义的模型定义使用 ResNet50 backbone
- 下载的预训练权重基于 MobileNetV2 backbone
- `load_state_dict()` 报错，key 完全不匹配

**解决方案**：

- 替换为 MODNet 官方模型定义（MobileNetV2 + IBNorm + SEBlock + LR/HR/Fusion 三分支）
- 写入 `algorithms/modnet/model.py`

### 2.12 MODNet DataParallel 权重前缀

**问题描述**：

- 预训练权重的 key 带有 `module.` 前缀（DataParallel 保存格式）
- 在 CPU 模式下直接 `load_state_dict()` 报错：key 不匹配

**解决方案**：

在 `algorithms/modnet/segmenter.py` 中加载权重时去除前缀：

```python
if not isinstance(self.model, nn.DataParallel):
    weights = {k.replace('module.', ''): v for k, v in weights.items()}
```

### 2.13 MODNet 推理质量差（人物半透明）

**问题描述**：

- 背景替换后人物出现严重半透明，背景穿透头部和身体
- 头顶灯光区域被误判为前景

**根本原因**：

- 输入图像强制 resize 到 512×512 正方形，破坏宽高比
- 输入尺寸未对齐到 32 的倍数（MODNet encoder 多次下采样要求）
- 在缩小后的图上做混合再放大，质量损失

**解决方案**（`algorithms/modnet/segmenter.py`）：

- 保持原始宽高比，按短边缩放到 `ref_size`
- 宽高对齐到 32 的倍数：`new_h = ((new_h - 1) // 32 + 1) * 32`
- 推理后将 matte resize 回原始分辨率，在原图上混合
- 通过 debug matte 可视化发现：灯光区域被误判为前景（白色），而非 matte 半透明

### 2.14 MODNet 速度慢和残影

**问题描述**：

- CPU 上帧率极低（~10 FPS），画面有拖影

**原因分析**：

- `ref_size=512` 推理太慢
- 时序平滑混入 70% 上一帧 alpha，移动时产生残影

**解决方案**：

- 将 `ref_size` 降到 384（速度和质量折中）
- 关闭时序一致性：`use_temporal_consistency = False`
- 简化 matte 后处理，移除耗时的形态学运算

### 2.15 RVM (RobustVideoMatting) 集成

**问题描述**：

- MODNet 在 CPU 上速度不理想，且对光照敏感
- 需要一个更适合实时视频的分割算法

**解决方案**：

- 集成 RVM 算法，使用 ONNX Runtime 推理
- 模型：`rvm_mobilenetv3_fp32.onnx`（14.3 MB）
- RVM 内置循环状态（recurrent states），天然具有时序一致性，无需额外平滑
- CPU 上推理耗时仅 24ms/帧，预计 FPS 40+

**新增文件**：

- `algorithms/rvm/__init__.py`
- `algorithms/rvm/segmenter.py`
- `models/rvm_mobilenetv3_fp32.onnx`

**当前默认算法**：直播压力测试后，上位机默认选择 RVM，界面仍支持切换 8 种算法：

```python
OPTIMAL_ALGORITHM_ID = 2
OPTIMAL_ALGORITHM_NAME = "RVM"
```

显式传入旧算法 ID 仍可通过 `BackgroundChangerFactory` 复现实验；若 RVM 初始化失败，运行链路回退到 MOG2，保证无模型环境下仍能启动基础背景替换。

## 3. 最终架构改进

### 3.1 核心类设计

#### BackgroundChangerFactory 类（`algorithms/factory.py`）

- 工厂模式，默认根据 `OPTIMAL_ALGORITHM_ID` 实例化 RVM，也允许实验脚本显式传入算法 ID
- 延迟导入避免 DLL 冲突
- RVM 初始化失败时回退到 MOG2

#### 三种分割器

| 类 | 文件 | 依赖 | 速度 |
| --- | --- | --- | --- |
| `RVMBackgroundChanger` | `algorithms/rvm/segmenter.py` | onnxruntime | 默认，直播综合最佳 |
| `MODNetBackgroundChanger` | `algorithms/modnet/segmenter.py` | torch | 质量高，CPU 慢 |
| `BackgroundChanger` | `algorithms/mediapipe/segmenter.py` | mediapipe | 历史对比 |
| `CVClassicBackgroundChanger` | `algorithms/cv_classic/segmenter.py` | opencv-python | 无模型基线 |

### 3.2 双策略可靠性保障

| 功能     | 主策略         | 备选策略       | 转换逻辑  |
| -------- | -------------- | -------------- | --------- |
| 图像加载 | OpenCV imread  | PIL Image.open | RGB ↔ BGR |
| 图像保存 | OpenCV imwrite | PIL Image.save | BGR → RGB |


## 4. 性能与兼容性

### 4.1 兼容性保障

- **OpenCV版本**：支持4.5.0+（即使缺少编解码器也能工作）
- **Python版本**：3.8+
- **操作系统**：Windows/Linux/macOS
- **图像格式**：JPG/JPEG/PNG/BMP

### 4.2 性能优化

- 多线程架构：视频处理在独立线程运行，避免UI卡顿

- MODNet 可调 `ref_size`（256/384/512）平衡速度和质量
